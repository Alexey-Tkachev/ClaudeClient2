# Claude Opus Desktop Client

Легковесный desktop-клиент для работы с Claude Opus из России без VPN и иностранных карт. Работает через API-ключ любого РФ агрегатора. Предоставляет минималистичный интерфейс для общения с моделью, отправки и получения текстовых файлов, включая архивы, и сохранения изменённых файлов на диск.


## Зачем нужен

Claude Opus Desktop Client нужен для работы с Claude из России.

## Для кого

Кто хочет работать с Claude Opus текстом и файлами.
Полезен будет для Java-разработчиков, которые хотят использовать Claude Opus как помощника по коду без браузерного интерфейса:

- отправлять модели несколько файлов проекта;
- получать исправленные классы обратно в структуре пакетов;
- вести несколько независимых чатов с разной историей и разными наборами файлов;
- чётко контролировать расход токенов и денег.

## Преимущества

По сравнению с обычным браузерным клиентом:

- не требует большого объёма оперативной памяти;
- не требует сложной настройки: вставил API-ключ — и работай;
- предоставляет удобный интерфейс работы с файлами и архивами;
- собирает файлы в специальных file-блоках (не нужно их искать по всей переписке);
- каждый чат хранит свои сообщения, входящие и исходящие файлы, настройки модели и статистику токенов;
- удобно отображает:
  - токены последнего запроса
  - токены всей сессии текущего чата
  - крайнюю ошибку (если была)
  - баланс на счёте агрегатора (при наличии у ключа права запроса баланса);
- файлы и история чатов сохраняются в локальной директории пользователя;
- сохраняет Java-классы в ZIP-архив с сохранением структуры пакетов;
- история чата не перегружается длинными вставками файлов;
- работает локально как Java Swing-приложение — не тормозит!

По сравнению с IDE-плагинами:

- Клиент проще и легче;
- не привязан к конкретной IDE;
- не требует настройки workspace-интеграций;
- можно использовать как отдельный рабочий инструмент рядом с IDE.

## Основной сценарий работы

1. Получить API-ключ любого РФ агрегатора LLM (например proxyapi.ru).
2. Запустить приложение.
3. Открыть вкладку **Настройки**.
4. Вставить API-ключ.
5. Нажать **Сохранить настройки**.
6. На вкладке **Файлы** нажать **Добавить файлы**.
7. Нажать **Отправить файлы**.
8. Перейти в чат, написать задачу и нажать **Отправить запрос**.
9. Если модель вернула файлы, они появятся в области **Файлы ИИ**.
10. Нажать **Загрузить файлы** для сохранения результата в `Downloads`.

## Скачать и запустить

Готовый JAR-файл доступен на странице Releases:

https://github.com/Alexey-Tkachev/ClaudeClient2/releases

Для запуска требуется Java 17 или новее.

Команда запуска:

```bash
java -jar ClaudeClient2.jar
```

## Техническая спецификация

### Платформа

- Java Swing desktop application.
- Maven-проект.
- Основной API-клиент: Anthropic Java SDK.
- Дополнительные HTTP-запросы к агрегатору выполняются отдельным сервисным классом.

### Модель данных чата

Каждый чат хранит собственное состояние:

- название чата;
- историю сообщений;
- модель;
- system prompt;
- `max_tokens` ответа;
- папку проекта;
- корень исходников;
- локальные файлы;
- Uploaded-файлы;
- Файлы ИИ;
- токены последнего запроса;
- суммарные токены сессии;
- последнюю ошибку.

Это означает, что новый чат не наследует контекст, файлы и историю другого чата.

### Работа с файлами

Пользовательский интерфейс файлов отделён от транспорта файлов.

Сейчас используется текстовый транспорт:

- локальные файлы читаются клиентом;
- поддерживаемые форматы файлов: `.java`, `.md`, `.txt`, `.xml`, `.json`, `.properties`, `.yml`, `.yaml`, `.sql`, `.gradle`, `.html`, `.css`, `.js`, `.ts` и другие текстовые файлы, а также архивы `.zip` с такими файлами;
- содержимое файлов скрыто дописывается к API-сообщению пользователя;
- в истории чата показывается только краткая сводка файлов;
- модель получает инструкцию возвращать файлы в формате:

```text
```file:relative/path/FileName.ext
<полное содержимое файла>
```
```

- клиент извлекает такие блоки из ответа;
- длинные тексты файлов удаляются из видимой истории;
- полученные файлы отображаются в области **Файлы ИИ**.

Архивы `.zip` раскрываются локально. Поддерживаемые текстовые файлы из архива передаются модели как отдельные файлы с сохранением относительных путей.

### Сохранение файлов ИИ

Если модель возвращает один файл без вложенного пути, клиент может сохранить его как отдельный файл.

Если файлов несколько или есть структура папок, клиент сохраняет ZIP-архив.

Для Java-классов клиент анализирует строку:

```java
package ru.example.demo;
```

и кладёт файл в архив по пути:

```text
<корень_исходников>/ru/example/demo/ClassName.java
```

Например, если в настройках чата выбран корень исходников `src/main/java`, структура папок в архиве будет:

```text
src/main/java/ru/example/demo/ClassName.java
```

### Хранение настроек и истории

Клиент использует пользовательскую директорию настроек. По умолчанию:

```text
~/.claude-opus-client
```

В ней хранятся:

- `config.properties` — API-ключ, base URL и путь к директории настроек;
- `chats.ser` — сериализованное состояние чатов для восстановления приложения;
- `chats-md/` — читаемые Markdown-снимки историй чатов;
- `history-index.txt` — индекс автосохранённых Markdown-файлов.

Файл `chats.ser` является внутренним бинарным файлом Java-сериализации. Он нужен приложению для точного восстановления состояния, включая списки файлов и настройки чатов. Его не предполагается редактировать вручную.

### Автосохранение

Чаты сохраняются автоматически:

- при отправке запроса;
- при получении ответа;
- при добавлении и отправке файлов;
- при переименовании чата;
- при удалении чата;
- при сохранении настроек.

Отдельная кнопка ручного сохранения всех чатов не требуется.

### Настройки модели

Модель привязана к конкретному чату. Для нового пустого чата модель можно выбрать из combo box. После появления истории чат считается привязанным к выбранной модели, и поле модели становится недоступным для редактирования.

Сейчас список моделей содержит только:

```text
claude-opus-4-7
```

### Токены и баланс

Клиент отображает:

- input/output tokens последнего запроса;
- input/output tokens всей сессии текущего чата;
- последнюю ошибку текущего чата;
- баланс агрегатора, если endpoint баланса доступен и API-ключ имеет соответствующее разрешение.

`max_tokens ответа` — верхняя граница ответа модели. Слишком большое значение может приводить к отказу агрегатора 402 из-за предварительного резервирования стоимости ответа.

## Ограничения

- Пока поддерживается только Anthropic Claude Opus 4.7.
- Проверялось только с агрегатором ProxyAPI.ru.
- Поддержка Anthropic beta Files API и `file_id` заложена архитектурно, но в текущей реализации отложена.
- Текущий файловый транспорт передаёт файлы как текст внутри сообщения, поэтому бинарные файлы напрямую не анализируются.
- Офисные документы и PDF не обрабатываются как полноценные документы; для корректной работы их содержимое рекомендуется предварительно экспортировать в текстовый формат.
- Для `.zip` из архива извлекаются только поддерживаемые текстовые файлы.
- Для очень больших проектов нужно контролировать объём отправляемых файлов и стоимость input tokens.
- UI ориентирован на desktop-использование и пока не является web-приложением.
- Не реализовано совместное редактирование файлов напрямую в проектной папке: клиент сохраняет результаты в `Downloads`, после чего пользователь сам переносит их в проект.

## Возможные направления развития

- Добавить транспорт через Anthropic Files API, если используемый API-провайдер поддерживает `/v1/files`.
- Добавить прямое применение patch/replace в папку проекта.
- Добавить поддержку Anthropic beta Files API для работы с PDF, а также DOCX/XLSX.
- Добавить подсветку чатов, превышающих половину контекстного окна Модели
- Добавить поддержку нескольких моделей.
- Добавить экспорт/импорт чатов.
- Добавить более точную оценку стоимости запроса до отправки.
- Значительно облегчить Клиента за счёт отказа от подключаемых библиотек в пользу самописных
